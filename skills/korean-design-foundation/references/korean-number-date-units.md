# 숫자·날짜·단위 표기 + 동적 조사

한국어 UI에서 숫자·통화·날짜·시간·단위를 표기하는 정본 규칙. `korean-content-design`(카피), `korean-dataviz`(표·KPI·차트), `korean-design-checks`(하드코딩 포맷 검출)가 이 파일을 근거로 판단한다.

**근거 출처**: 국립국어원 한글 맞춤법 제43항(단위 명사 띄어쓰기) · `korean-skills:style-guide` 스킬 `references/numbering-units.md`(패턴 7·8·9), `references/datetime-reference.md`(패턴 16·17) · ISO 8601 · 본 스킬 `type-scale.md`(tabular-nums).

---

## 1. 만/억 그루핑

한국어 숫자는 4자리(만·억) 단위로 끊어 읽지만, 콤마는 3자리(천 단위)로 찍는다. 이 불일치 때문에 **표면별로 축약 여부를 먼저 정한다.**

### 표면별 축약 선택

| 표면 | 표기 | 예 (1,250,000) | 근거 |
|---|---|---|---|
| 대시보드 KPI | **축약** (만/억) | 125만 | 스캔 속도 우선. 정확값은 툴팁·보조 텍스트로 제공 |
| 표(테이블) | **원값 + 콤마** | 1,250,000 | 열 내 비교·합산 검증이 목적. 자릿수 절약이 필요하면 열 헤더에 `(단위: 만 원)` 명시 후 전체 스케일링 |
| 본문·카피 | 축약 (읽는 흐름) | 125만 원 | "가격은 1,250,000원" 보다 "가격은 125만 원"이 자연스럽다 |
| 정산·회계·영수증 | **축약 금지** | 1,250,000원 | 법적·회계 정확성. 반올림 표시 불가 |
| 차트 축 라벨 | 축약 | 100만 / 120만 | 축 라벨은 3~5자 이내 (`korean-dataviz` 축 규칙) |

한 화면 안에서 축약과 원값을 혼용하지 않는다. 단, "KPI=축약 + 상세 표=원값" 조합은 표면이 다르므로 허용 (`style-guide` numbering-units.md 패턴 9: 동일 문맥 내 `10,000원`↔`5만 원` 혼용 금지).

### 변환 예시표

| 원값 | 축약 표기 | 비고 |
|---|---|---|
| 3,000 | 3,000 | **1만 미만은 축약하지 않는다** |
| 10,000 | 1만 | |
| 99,999 | 10만 | 반올림 (정확값 필요 시 원값 유지) |
| 1,250,000 | 125만 | |
| 12,540,000 | 1,254만 | 4자리까지는 만 단위 유지 |
| 125,000,000 | 1.25억 | 억 전환. 소수 최대 2자리 |
| 1,250,000,000 | 12.5억 | |
| 125,000,000,000 | 1,250억 | 조 미만은 억 단위 유지 |
| 1,500,000,000,000 | 1.5조 | |

- 만 → 억 전환점: **1억(100,000,000) 이상**. "12,500만"처럼 만 단위 5자리 표기는 금지.
- 영문 축약(1.2M, 350K)은 한국어 표면에서 금지. 국제 공용 대시보드만 예외.

### 반올림 규칙

| 규칙 | 내용 |
|---|---|
| 유효숫자 | 축약값은 유효숫자 **3자리 이내**, 소수는 최대 1자리(억·조는 2자리) |
| 방식 | 사사오입(round half up) 일관 적용 |
| 정확값 접근 | 반올림으로 원값과 달라지면 툴팁·`title` 속성·보조 행으로 원값 제공. 화면에 "약"을 붙일지는 카피 어조에 따름 |
| % 합계 보정 | 비율 반올림 후 합이 100%가 안 되면 최대잔여법(largest remainder)으로 보정. 개별 값을 임의 조정하지 않는다 |
| 소수 자릿수 통일 | 같은 열·같은 KPI 그룹은 소수 자릿수를 통일 (12.5% / 3.0% — 3%가 아니라 3.0%) |

## 2. 통화

| 규칙 | 표기 | 예 |
|---|---|---|
| 기본 | 숫자 뒤에 `원` | 33,000원 |
| ₩ 기호 | 숫자 **앞** — 표 열·가격 태그 등 자리가 좁은 UI 한정 | ₩33,000 |
| 혼용 금지 | 한 표면에서 `원`과 `₩`를 섞지 않는다 | — |
| 축약 + 원 | 축약 시 `원`은 띄어 쓴다(단위 명사) | 125만 원 |
| 소수 | 원화는 소수점 없음. 환율·주가 등 필요할 때만 예외 | 1,350.50원 (환율) |

### 부가세 표기

| 상황 | 표기 | 예 |
|---|---|---|
| B2B 요금제·견적 | 가격 직후 괄호로 **부가세 별도** 명시 | 월 33,000원 (VAT 별도) |
| B2C 소비자가 | 부가세 **포함가** 표시가 기본 (표시가격 = 지불가격) | 9,900원 |
| 포함가임을 강조 | 괄호 병기 | 36,300원 (VAT 포함) |

부가세 문구 없이 B2B 가격만 노출하는 것은 분쟁 소지 — 요금제 화면에서는 반드시 병기 (`korean-web-layout` landing-trust-conversion 요금제 규칙과 연동).

### 음수·증감 표기

| 맥락 | 표기 | 예 |
|---|---|---|
| 회계·정산 음수 | 괄호 표기 | (1,200,000) |
| 일반 UI 음수 | 마이너스 부호 | -1,200,000원 |
| 증감(델타) | 부호 **항상 병기** (+ / -) | +12.5% / -3.2% |
| 색 규칙 | 색만으로 증감 구분 금지 — 부호·아이콘 병행 (WCAG 1.4.1) | ▲ +2.1% |
| 금융 관례 | 국내 증권·금융은 **상승=빨강, 하락=파랑** (글로벌과 반대). 도메인 확인 후 적용, 색값은 `color-contrast.md` | — |

한 화면에서 괄호 음수와 부호 음수를 혼용하지 않는다.

## 3. 단위 붙임 (띄어쓰기)

국립국어원 원칙과 UI 관례가 갈리는 지점. **어느 쪽을 따르는지 표면별로 고정**하고 혼용하지 않는 것이 규칙의 핵심이다.

### 국립국어원 관례 (문서·공문·제안서 본문)

한글 맞춤법 제43항: 단위 명사는 띄어 씀이 원칙, **아라비아 숫자 뒤에서는 붙여 씀을 허용**.

| 구분 | 원칙 | 허용 | 비고 |
|---|---|---|---|
| 한글 수사 + 단위 | 세 명, 두 건 | — | 항상 띄어쓰기 |
| 아라비아 숫자 + 단위 | 3 명, 5 건 | 3명, 5건 | 실무 문서는 허용형(붙임)이 우세 |
| 축약 수치 + 단위 명사 | 125만 원, 3억 명 | — | `만/억`과 `원/명` 사이는 띄운다 |

### UI 관례 (웹앱·대시보드·모바일)

| 단위 | 표기 | 예 | 근거 |
|---|---|---|---|
| 원·명·건·개·회 | **붙여쓰기 통일** | 3,200원 · 15명 · 42건 · 8개 · 3회 | 줄바꿈 분리 방지, 스캔 속도 (`style-guide` numbering-units.md 패턴 8: 의존명사 단위 붙여쓰기) |
| % | **붙여쓰기** (문서·UI 공통) | 12.5% | `50 %` 금지 |
| SI 단위 (kg, km, GB) | UI는 붙임 허용, 기술 문서는 SI 원칙(띄움) | 2.5GB (UI) / 2.5 GB (기술 문서) | 국제단위계 표기법 |
| 축약 + 단위 | `만/억` 뒤 단위 명사는 UI에서도 띄운다 | 125만 원, 1.2억 건 | "125만원"은 허용 오차지만 정본은 띄움 |

- 숫자와 단위가 줄 끝에서 분리되지 않게 한다 — 붙여쓰기로 대부분 해결되며, 띄우는 경우(`125만 원`)는 `&nbsp;` 또는 `white-space: nowrap` 래핑.
- 범위: `3~5건`, `20~25명` — 물결표 앞뒤 붙임, 단위는 뒤에 한 번만.

## 4. 날짜

### 표면별 선택표

| 표면 | 형식 | 예 | 근거 |
|---|---|---|---|
| 문서·제안서·공문·계약 | `YYYY년 M월 D일` | 2026년 7월 3일 | 공문서 관행 (`style-guide` datetime-reference.md 패턴 16). **0 패딩 없음** — "07월 03일" 금지 |
| UI 컴팩트 (표·리스트·메타) | `YYYY.MM.DD` | 2026.07.03 | 자릿수 고정 정렬. **2자리 0 패딩 고정**, 끝에 마침표 없음 |
| 국제·데이터·API·로그 | `YYYY-MM-DD` (ISO 8601) | 2026-07-03 | 정렬·파싱·다국어 안전 |
| 같은 해 반복 노출 | `M월 D일` 또는 `MM.DD` | 7월 3일 / 07.03 | 연도 생략은 같은 해가 명백할 때만 |

- 슬래시(`7/3`, `07/03/2026`)는 국가별 해석이 갈려 **비권장** (datetime-reference.md 패턴 16).
- `Intl.DateTimeFormat('ko-KR')` 기본 출력은 `2026. 7. 3.`(공백+끝 마침표)로 위 정본과 다르다 — UI 컴팩트 형식은 직접 포맷팅한다.
- 기간: `2026.07.01 ~ 2026.07.31` (물결표 앞뒤 공백), 같은 달이면 `7월 1일~31일`.

### 요일 표기

| 표면 | 표기 | 예 |
|---|---|---|
| UI (예약·일정·리스트) | 괄호 한 글자 | 2026.07.03(금) |
| 문서·안내문 | 괄호 한 글자 또는 전체 | 2026년 7월 3일(금) / 7월 3일 금요일 |
| 요일이 판단 재료일 때 | 반드시 병기 | 배송·예약·마감 UI |

요일은 날짜 바로 뒤 괄호, 괄호 앞 공백 없음.

## 5. 시간

### 오전/오후 vs 24시간 선택 기준

| 기준 | 오전/오후 (12시간제) | 24시간제 |
|---|---|---|
| 대상 | 일반 소비자 (예약·일정·알림·마케팅) | 운영자·관리자·B2B 대시보드·로그·기술 문서 |
| 예 | 오후 3시 30분 / 오후 3:30 | 15:30 |
| 근거 | 일상 관행 (`style-guide` datetime-reference.md 패턴 17) | 모호성 제거, ISO 8601 |

- `AM/PM`과 `오전/오후` 혼용 금지 — 한국어 표면은 `오전/오후`만.
- 정오·자정 모호성: "오후 12시" 대신 **정오 / 자정** 또는 `12:00` / `00:00`.
- 분이 0이면 생략 가능: "오후 3시" (단, 같은 리스트 안에서는 `3:00 PM`식 자릿수 통일 우선).
- 시간대가 문제 되는 표면(글로벌·예약)은 `15:30 (KST)` 병기.

### 상대시간 경계

| 경과 시간 | 표기 | 비고 |
|---|---|---|
| 60초 미만 | 방금 전 | "0분 전" 금지 |
| 1~59분 | N분 전 | 3분 전 |
| 1~23시간 | N시간 전 | 5시간 전 |
| 달력상 전날 | 어제 | 경과 시간이 아니라 **달력일 기준**으로 전환 |
| 2일 전 ~ 같은 해 | M월 D일 | 상대 표기 종료 |
| 해가 다름 | YYYY.MM.DD | |

- **절대시간 접근 보장**: 상대시간은 툴팁·`title`·`<time datetime="...">`으로 절대시간을 항상 함께 제공한다.
- 미래 방향은 대칭: "N분 후 / N시간 후 / 내일".
- 상대시간은 화면 체류 중 갱신하거나(1분 간격), 갱신하지 않는다면 "방금 전" 경계를 보수적으로 잡는다.
- 법적·기록성 표면(결제 내역·감사 로그)은 상대시간 금지, 절대시간만.

## 6. tabular-nums 연계

`type-scale.md` §한·영·숫자 혼식 — "숫자는 표·KPI에서 **tabular-nums**(고정폭 숫자)로 정렬 안정화."

| 대상 | 규칙 |
|---|---|
| 표 숫자 열 | `font-variant-numeric: tabular-nums` + **오른쪽 정렬** (`korean-dataviz` TableWrapper 규칙과 세트) |
| KPI 수치·카운트다운·타이머 | tabular-nums 필수 — 값 변할 때 폭 흔들림(레이아웃 시프트) 방지 |
| 본문 속 숫자 | 기본 proportional 유지 — 전역 적용 금지 |

- **적용 범위 원칙 (F2)**: tabular-nums는 **표시형 숫자(표·KPI·성과 수치·타이머·입력값)에만** 건다. "숫자 전부 tabular" 같은 일괄 지시가 있어도 이 표가 우선한다 — 본문 프로즈 숫자까지 걸면 글자 리듬이 깨진다.
| Tailwind | `tabular-nums` 유틸리티 클래스 |
| 폰트 | Pretendard는 `tnum` 지원. 대체 폰트 폴백 시에도 유지되는지 검수 |

## 7. 동적 조사 헬퍼

변수(사용자명·파일명·플랜명)에 조사가 붙는 순간 `홍길동이 / 김서연가` 류 파손이 생긴다. 해법은 두 층: **①받침 판정 헬퍼 ②조사 회피 재작성**. 완성형 한글로 끝나지 않는 값(영문·숫자·이모지)이 올 수 있으면 ②를 우선한다.

### 조사 선택표

| 마지막 글자 | 은/는 | 이/가 | 을/를 | 와/과 | (으)로 |
|---|---|---|---|---|---|
| 받침 있음 | 은 | 이 | 을 | **과** | 으로 |
| 받침 있음 — **ㄹ 받침** | 은 | 이 | 을 | 과 | **로** ← 예외 |
| 받침 없음 | 는 | 가 | 를 | **와** | 로 |

- `와/과`는 방향이 반대다(받침 있음→과). 기계적으로 뒤집지 말 것.
- `(으)로`는 ㄹ 받침이 유일한 예외: 서울**로**(ㄹ받침), 부산**으로**(ㄴ받침), 제주**로**(받침 없음).

### TypeScript 헬퍼

```ts
const JOSA_TABLE = {
  '은/는': ['은', '는'],
  '이/가': ['이', '가'],
  '을/를': ['을', '를'],
  '과/와': ['과', '와'],
  '으로/로': ['으로', '로'],
} as const;

type JosaPair = keyof typeof JOSA_TABLE;

const RIEUL = 8; // 종성 인덱스 8 = ㄹ

/** 마지막 글자의 받침 정보. 완성형 한글(가-힣)이 아니면 null */
function batchimOf(word: string): { has: boolean; isRieul: boolean } | null {
  const ch = word.trimEnd().at(-1);
  if (!ch) return null;
  const code = ch.charCodeAt(0);
  if (code < 0xac00 || code > 0xd7a3) return null; // 한글 음절 밖
  const jong = (code - 0xac00) % 28; // 0이면 받침 없음
  return { has: jong > 0, isRieul: jong === RIEUL };
}

/**
 * josa('홍길동', '이/가')  → '홍길동이'  (ㅇ받침)
 * josa('이수지', '이/가')  → '이수지가'  (받침 없음)
 * josa('서울', '으로/로')  → '서울로'    (ㄹ받침 예외)
 * josa('부산', '으로/로')  → '부산으로'  (ㄴ받침)
 * josa('Pro', '을/를')     → 'Pro을(를)' (판정 불가 → 병기 폴백)
 * josa('Pro', '으로/로')   → 'Pro(으)로' (병기 폴백 — 이 쌍만 관행 표기가 다름)
 */
export function josa(word: string, pair: JosaPair): string {
  const [withBatchim, withoutBatchim] = JOSA_TABLE[pair];
  const b = batchimOf(word);
  if (b === null) {
    // 폴백: 병기 — '으로/로'만 관행 표기가 '(으)로'로 다름 (예: 'Pro(으)로')
    return pair === '으로/로' ? `${word}(으)로` : `${word}${withBatchim}(${withoutBatchim})`;
  }
  if (pair === '으로/로') return word + (b.has && !b.isRieul ? '으로' : '로');
  return word + (b.has ? withBatchim : withoutBatchim);
}
```

- 받침 판정 정본: `(code - 0xAC00) % 28 > 0`.
- `으로/로`는 반드시 ㄹ 받침 예외를 태운다 — 이 예외가 빠진 구현이 가장 흔한 결함.
- 병기 폴백(`을(를)`)은 최후 수단이다. 최종 카피에 병기가 남아 있으면 검수 실패로 본다(`korean-design-checks` 검출 대상) — 아래 회피 재작성을 먼저 시도한다.

### 조사 회피 재작성표

변수 뒤에 조사를 붙이지 말고 **문장을 재구성**해 조사를 고정 명사에만 붙인다.

| 패턴 | 위험한 원문 | 재작성 |
|---|---|---|
| 콜론 후치 | `{name}이(가) 삭제됩니다` | `다음 항목이 삭제됩니다: {name}` |
| 고정 명사 수식 | `{file}을(를) 업로드했습니다` | `{file} 파일을 업로드했습니다` — 조사는 고정 명사 "파일"에 붙는다 |
| 분류어 후치 | `{team}와(과) 공유합니다` | `공유 대상: {team}` |
| 단위 결합 | `{n}이(가) 처리됨` | `{n}건이 처리되었습니다` — 숫자 뒤 단위(건)가 조사를 받는다 |
| 라벨식 축약 | `{name}이(가) 저장되었습니다` | `{name} 저장됨` — 짧은 토스트·상태 라벨은 조사 생략 |
| 어순 교체 | `{plan}으로 변경할까요?` | `요금제를 {plan}(으)로 변경할까요?` → `변경할 요금제: {plan}` |

- 따옴표로 감싸는 것(`'{name}'을(를)`)은 조사 문제를 해소하지 못한다 — 재작성이 정답.
- 재작성이 불가능한 자리(자연스러운 문장이 필수인 온보딩·마케팅 카피)에만 `josa()` 헬퍼를 쓰고, 변수 도메인이 한글로 보장되는지 확인한다.
