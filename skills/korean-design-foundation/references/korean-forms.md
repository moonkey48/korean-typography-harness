# 한국 폼 관례

한국 서비스 폼의 도메인 필드 규칙. 각 항목은 **구조 규칙 → 마크업 힌트 → 검증 실패 카피** 순서로 판정 가능하게 적는다. 규칙마다 근거를 붙이고, 근거가 제도·표준이 아니면 '관례'로 표기한다. 카피 어조·조사 처리의 일반 원칙은 `korean-content-design`의 korean-ux-writing을 따른다.

## 공통 원칙

| 규칙 | 내용 | 근거 |
|---|---|---|
| 검증 타이밍 | blur 시 1차 + submit 시 전체. 타이핑 중 실시간 오류 표시는 비밀번호 조건 체크리스트처럼 "충족 표시"에만 사용 | 입력 도중 오류 노출은 미완성 입력을 오류로 오판 — 관례 |
| blur의 범위 | **blur는 형식 검증만** — 빈 필드(필수 미입력)는 blur에서 벌하지 않고 **submit에서만** 오류 처리 | 훑어보기만 한 필드를 오류로 물들이면 시작 전에 겁먹는다 — 관례 (F3, 파일럿 확정) |
| 오류 카피 구조 | ①무엇이 잘못됐는지 ②(알면) 원인 ③다음 행동. 비난·막다른 길 금지 | korean-ux-writing 오류 구조 |
| 입력 보존 | 검증 실패해도 입력값을 지우지 않는다. 첫 오류 필드로 focus 이동 | 사용성 휴리스틱(오류 회복) — usability-heuristics.md 참조 |
| 표시값 ≠ 저장값 | 하이픈·마스크는 표시용. 저장·전송은 숫자만 | 포맷 변경 내성 — 관례 |
| 라벨 항상 표시 | placeholder를 라벨 대용으로 쓰지 않는다. 규칙은 helper text에 사전 노출 | 입력 시작하면 placeholder 소실 — WCAG 3.3.2 |

## 1. 휴대폰번호

### 구조 규칙

| 규칙 | 내용 | 근거 |
|---|---|---|
| 자리수 | 010 기준 **3-4-4** (총 11자리). 010-XXXX-XXXX | 2004년 010 번호통합 이후 신규 번호는 전부 010 11자리 — 잔여 01X는 2G 종료(SKT 2020, LG U+ 2021)와 함께 소멸 |
| 입력 방식 | **단일 입력 필드 + 자동 하이픈** 삽입 | 토스·네이버·카카오 주류 패턴 — 관례 |
| 3분할 지양 | 앞/중간/뒤 3개 필드로 쪼개지 않는다 | ①자동 포커스 이동이 수정(백스페이스) 시 역이동에 실패 ②`010-1234-5678` 통째 붙여넣기 깨짐 ③`autocomplete="tel"` 자동완성이 단일 필드에만 매핑 ④스크린리더가 하나의 번호로 읽지 못함 |
| 자동 하이픈 | 숫자만 남기고 3-4-4로 재조립해 표시. 커서 위치 보존 처리 필수 | 청킹 가독성 — 관례 |
| 저장 형식 | 숫자 11자리(`01012345678`) | 표시 포맷과 분리 — 관례 |

### 마크업 힌트

```html
<input type="tel" inputmode="numeric" autocomplete="tel"
       maxlength="13" placeholder="010-0000-0000">
```

| 속성 | 값 | 이유 |
|---|---|---|
| `type` | `tel` | 모바일 전화 키패드 + 시맨틱 |
| `inputmode` | `numeric` | 숫자 키패드 강제(일부 브라우저에서 tel 키패드에 기호 포함) |
| `autocomplete` | `tel` (국내 전용이면 `tel-national`) | 브라우저 자동완성 연결 |
| `maxlength` | `13` | 하이픈 포함 최대 길이 |
| 제출 검증 | `^010-\d{4}-\d{4}$` (하이픈 제거 후 `^010\d{8}$`) | 형식 검증 |

### 인증번호(SMS OTP) 입력

| 규칙 | 내용 | 근거 |
|---|---|---|
| 자리수 | **6자리** 숫자 단일 필드 | 국내 SMS 인증 표준 자리수 — 관례 |
| 타이머 | **3:00** 카운트다운을 입력 필드 안 또는 우측에 표시. 만료 시 입력 잠그지 말고 재전송 유도 | 통신사 SMS 인증 유효시간 3분 — 관례 |
| 재전송 | 만료 전에도 허용하되 쿨다운(예: 30초)·일 횟수 제한. 재전송 시 이전 번호 무효화 안내 | 오발송·미수신 대응 — 관례 |
| 자동 채움 | `autocomplete="one-time-code"` 로 OS의 SMS 자동 입력 연결 | WHATWG autocomplete 명세, iOS/Android 지원 |

```html
<input type="text" inputmode="numeric" autocomplete="one-time-code"
       maxlength="6" pattern="\d{6}">
```

재전송 카피: `인증번호를 다시 보냈어요. 1분 안에 문자가 오지 않으면 스팸함을 확인해 주세요.`

### 검증 실패 카피 (원인+해결)

| 상황 | 카피 |
|---|---|
| 형식 오류 | 휴대폰 번호 형식이 올바르지 않아요. 010으로 시작하는 숫자 11자리를 입력해 주세요. |
| 인증번호 불일치 | 인증번호가 일치하지 않아요. 문자로 받은 6자리 숫자를 다시 확인해 주세요. |
| 인증번호 만료 | 인증번호 유효 시간이 지났어요. 재전송을 눌러 새 인증번호를 받아 주세요. |
| 재전송 제한 | 인증번호 요청이 너무 많아요. 1분 뒤에 다시 시도해 주세요. |

## 2. 주소

### 구조 규칙 — 다음(카카오) 우편번호 서비스 임베드

| 단계 | 동작 | UI |
|---|---|---|
| 1 | 사용자가 [우편번호 찾기] 버튼 클릭 | 우편번호·기본주소 필드는 직접 입력 불가(readonly) |
| 2 | 다음 우편번호 서비스 레이어/임베드 열림 → 동·건물명·도로명으로 검색 | 모바일은 전체 화면 레이어 — 관례 |
| 3 | 결과 선택 → 우편번호(5자리)·**도로명/지번 주소 자동 입력** | 사용자가 도로명/지번 중 선택 가능 |
| 4 | **상세주소(동·호수)만 수기 입력** — 선택 직후 상세주소 필드로 focus 이동 | 다음 행동 안내 — 관례 |

| 규칙 | 내용 | 근거 |
|---|---|---|
| 자유입력 금지 | 기본주소를 텍스트 필드로 받지 않는다 | ①표기 변형(동/리/번지/건물명)으로 표준화 불가 ②2015년 우편번호 5자리 개편 이후 수기 매칭 불가 ③배송·청구지 오류 비용 직결 |
| 다음 우편번호 채택 | 사실상 표준. 무료·API 키 불필요·행안부 도로명주소 DB 자동 갱신 | 관례(국내 커머스·핀테크 대부분 채택) |
| 저장 필드 | 우편번호 / 도로명주소 / 지번주소 / 상세주소 / (참고항목) 분리 저장 | 서비스 응답 필드 구조 그대로 — 관례 |

### 마크업 힌트

```html
<input name="postcode" readonly autocomplete="postal-code">
<input name="address"  readonly autocomplete="address-line1">
<input name="addressDetail" autocomplete="address-line2"
       placeholder="상세주소 (동/호수)">
```

- readonly 필드도 탭 순서에 두고, 값 변경은 [우편번호 찾기] 재실행으로만.
- 상세주소는 선택 입력 허용(단독주택 등 상세주소 없는 경우) — 필수로 강제하지 않는다. 근거: 관례.

### 검증 실패 카피 (원인+해결)

| 상황 | 카피 |
|---|---|
| 주소 미선택 제출 | 주소를 아직 선택하지 않았어요. 우편번호 찾기를 눌러 주소를 검색해 주세요. |
| 상세주소 필수 서비스 | 상세주소가 비어 있어요. 동·호수 등 나머지 주소를 입력해 주세요. |

## 3. 사업자등록번호

### 구조 규칙

| 규칙 | 내용 | 근거 |
|---|---|---|
| 자리수 | 10자리 **3-2-5** (XXX-XX-XXXXX) | 국세청 부여 체계: 앞 3자리 세무서 코드, 가운데 2자리 개인/법인 구분, 뒤 5자리 일련번호 4 + 검증 1 |
| 입력 방식 | 단일 필드 + 자동 하이픈 마스크(표시용). 저장은 숫자 10자리 | 휴대폰번호와 동일 원칙 — 관례 |
| 1차 검증 | 클라이언트에서 **체크섬** 검증 → 오타 즉시 차단 | 국세청 공개 검증 알고리즘 |
| 2차 검증 | 체크섬 통과 ≠ 실존 사업자. 국세청 진위확인 API(공공데이터포털 사업자등록정보 진위확인·상태조회)는 서버에서 별도 호출 | 체크섬은 형식 검증일 뿐 휴·폐업 여부를 모름 |

### 체크섬 알고리즘

가중치 `1,3,7,1,3,7,1,3,5`를 앞 9자리에 곱해 합산하고, 9번째 자리×5의 십의 자리를 더한 뒤, 10의 보수가 마지막 자리와 같으면 유효.

```js
function isValidBizno(value) {
  const d = value.replace(/\D/g, "");
  if (!/^\d{10}$/.test(d)) return false;
  const w = [1, 3, 7, 1, 3, 7, 1, 3, 5];
  let sum = 0;
  for (let i = 0; i < 9; i += 1) sum += Number(d[i]) * w[i];
  sum += Math.floor((Number(d[8]) * 5) / 10);
  return (10 - (sum % 10)) % 10 === Number(d[9]);
}
```

### 마크업 힌트

```html
<input type="text" inputmode="numeric" autocomplete="off"
       maxlength="12" placeholder="000-00-00000">
```

- `autocomplete="off"`: 매핑되는 표준 토큰이 없다. 브라우저가 무관한 값을 채우는 것을 막는다.
- 제출 검증: 하이픈 제거 후 `^\d{10}$` + 체크섬.

### 검증 실패 카피 (원인+해결)

| 상황 | 카피 |
|---|---|
| 자리수 부족 | 사업자등록번호는 숫자 10자리예요. 빠진 자리가 없는지 확인해 주세요. |
| 체크섬 실패 | 사업자등록번호가 유효하지 않아요. 사업자등록증의 번호와 같은지 확인해 주세요. |
| 진위확인 실패(서버) | 국세청에서 확인되지 않는 번호예요. 휴·폐업 여부를 확인하거나 다른 번호를 입력해 주세요. |

## 4. 생년월일

### 구조 규칙 — 8자리 단일 입력 vs 3-select

| 기준 | 8자리 단일 입력 (권장) | 연/월/일 3-select |
|---|---|---|
| 입력 속도 | 숫자 키패드 연속 8타 | 스크롤 3회 — 연도 목록이 특히 길다 |
| 모바일 | `inputmode="numeric"` 키패드 유지 | 네이티브 picker가 OS마다 제각각 |
| 유효성 | 실존 날짜 검증 필요(0231 등) — 코드 1곳 | 조합 오류(2월 31일) 별도 차단 필요 |
| 접근성 | 라벨 1개, `autocomplete="bday"` 연결 | select 3개의 라벨·읽기 순서 부담 |
| 채택 | 토스 등 금융·핀테크 주류 — 관례 | 구형 회원가입에 잔존 |

- 권장: **단일 입력 8자리(YYYYMMDD)** + 입력 완료 시 `1990.01.01` 형식으로 표시 정리. 근거: 위 비교표 + 관례.
- 3-select를 쓴다면 `autocomplete`를 `bday-year` / `bday-month` / `bday-day`로 분리 매핑.

### 만 나이 표기

| 규칙 | 내용 | 근거 |
|---|---|---|
| 기준 | 나이 계산·표기는 **만 나이 단일 기준** | 2023-06-28 만 나이 통일법(행정기본법·민법 개정) 시행 |
| 표기 | 전환기 혼동 방지를 위해 "만 34세" 병기가 안전. 법정 연령 제한(만 14세 미만 가입 등)은 반드시 '만' 명시 | 관례 |
| 계산 | 생일 도래 여부 반영: 올해 - 출생연도, 생일 전이면 -1 | 만 나이 정의 |

### 마크업 힌트

```html
<input type="text" inputmode="numeric" autocomplete="bday"
       maxlength="8" placeholder="YYYYMMDD">
```

- `type="date"`는 지양: 브라우저별 picker 편차가 크고 8자리 연속 입력이 안 된다 — 관례.

### 검증 실패 카피 (원인+해결)

| 상황 | 카피 |
|---|---|
| 형식 오류 | 생년월일 형식이 올바르지 않아요. 19900101처럼 숫자 8자리로 입력해 주세요. |
| 실존하지 않는 날짜 | 달력에 없는 날짜예요. 월과 일을 다시 확인해 주세요. |
| 연령 제한 | 만 14세 미만은 가입할 수 없어요. 보호자 동의 절차를 이용해 주세요. |

## 5. 본인인증 (PASS/SMS)

### 표준 플로우

| 단계 | 사용자 행동 | UI 상태 |
|---|---|---|
| 1 | 인증 수단 선택 (PASS 앱 / SMS / 기타) | 수단 버튼 목록. 본인확인기관(NICE·KCB·SCI) 표준창을 팝업/리다이렉트로 여는 방식이 관례 |
| 2 | 통신사 선택 (SKT / KT / LG U+ / 알뜰폰) | 알뜰폰은 망 사업자 하위 선택 |
| 3 | 이름·생년월일·성별·휴대폰번호 입력 + 약관 동의(개인정보·고유식별정보·통신사 이용·서비스 약관 4종) | 전체 동의 + 개별 보기 링크 |
| 4a | **PASS**: 앱 푸시 수신 → 앱에서 승인 | 대기 화면 + 폴링. "PASS 앱에서 인증을 완료해 주세요" + 미수신 시 SMS 전환 링크 |
| 4b | **SMS**: 6자리 인증번호 입력 | §1 인증번호 규칙 그대로(6자리·3분 타이머·재전송) |
| 5 | 인증 완료 → CI/DI 수신 → 원래 화면 복귀 | 성공 표시 후 자동 복귀. 인증된 이름·번호는 읽기 전용으로 잠금 — 관례 |

### UI 상태 정의

| 상태 | 화면 | 카피 방향 |
|---|---|---|
| idle | 수단 선택 대기 | — |
| requesting | 요청 전송 중 | 버튼 로딩 + 이중 제출 방지 |
| waiting | 승인/입력 대기 | 타이머 표시, 취소 경로 제공(막다른 길 금지) |
| success | 인증 완료 | 완료 표시 후 다음 단계 자동 진행 |
| failed | 인증 실패 | 원인+해결 카피, 입력값 보존, 재시도 버튼 |
| expired | 유효 시간 만료 | 재요청 유도 |

| 규칙 | 내용 | 근거 |
|---|---|---|
| 팝업 차단 대응 | 표준창이 팝업으로 열리므로 차단 감지 시 안내 카피 + 재시도 버튼 제공 | 관례 |
| 정보 불일치 안내 | 실패 최다 원인은 통신사 가입 정보와 입력 정보 불일치 — 실패 카피에 명시 | 관례 |

### 마크업 힌트

본인인증은 본인확인기관 표준창을 여는 **플로우형**이라 입력 대부분은 표준창 안에서 일어난다. 자체 화면에 남는 요소에만 아래 힌트를 적용한다.

```html
<input type="text" autocomplete="name">
<button type="button">본인인증 하기</button>
```

| 포인트 | 내용 | 이유 |
|---|---|---|
| 이름 필드 | `autocomplete="name"` | 브라우저 자동완성 연결 |
| 팝업 트리거 | `type="button"` 명시. `window.open` 반환값이 `null`이면 팝업 차단으로 판정 → 안내 카피 + 재시도 버튼 노출 | form 안에서 기본 `type="submit"` 오동작 방지 + 위 '팝업 차단 대응' 규칙의 구현 지점 |
| 생년월일·휴대폰번호 | 표준창 밖에서 자체 수집하는 경우 §4·§1 마크업 그대로 재사용 | 중복 정의 방지 — §1·§4가 정본 |
| SMS 폴백 | 6자리 인증번호 입력은 §1 '인증번호(SMS OTP) 입력' 마크업 재사용 | §1이 정본 |

### 검증 실패 카피 (원인+해결)

| 상황 | 카피 |
|---|---|
| 정보 불일치 | 본인인증을 완료하지 못했어요. 입력한 이름과 생년월일이 통신사 가입 정보와 같은지 확인해 주세요. |
| 팝업 차단 | 인증 창이 열리지 않았어요. 브라우저의 팝업 차단을 해제한 뒤 다시 시도해 주세요. |
| PASS 미승인 만료 | 인증 요청이 만료됐어요. PASS 앱 알림을 확인하기 어려우면 문자 인증으로 진행해 주세요. |

## 6. 이메일·비밀번호

### 구조 규칙

| 규칙 | 내용 | 근거 |
|---|---|---|
| 조건 사전 표시 | 비밀번호 규칙(길이·조합)은 helper text로 **입력 전에** 노출. 실패한 뒤에야 알려주는 방식 금지 | 오류 예방 휴리스틱 — usability-heuristics.md 참조 |
| 실시간 충족 표시 | 입력 중 조건 체크리스트(8자 이상 ✓, 숫자 포함 ✓ …)로 충족 항목만 표시. 미충족을 오류로 붉게 칠하지 않는다 | 토스·네이버 패턴 — 관례 |
| 보기 토글 | 비밀번호 필드에 표시/숨김 토글 버튼 제공 | 모바일 오타율 절감 — 관례 |
| 규칙 설계 | 길이 8자 이상 중심. 주기적 변경 강제·과도한 특수문자 규칙 지양 | NIST SP 800-63B |
| 확인 필드 | 보기 토글이 있으면 "비밀번호 확인" 재입력 필드는 생략 가능 | 관례(서비스 정책에 따름) |

### 마크업 힌트

```html
<input type="email" autocomplete="email" inputmode="email">

<input type="password" autocomplete="new-password"
       minlength="8" aria-describedby="pw-rules">
<button type="button" aria-label="비밀번호 표시" aria-pressed="false">…</button>
```

| 포인트 | 내용 | 이유 |
|---|---|---|
| `autocomplete` | 가입 `new-password`, 로그인 `current-password` | 브라우저 비밀번호 제안·자동완성 연결 |
| 보기 토글 | `type`을 `password↔text` 전환. `aria-pressed` 갱신, 전환 시 커서 위치 보존 | 접근성 + 입력 흐름 유지 |
| 이메일 검증 | `@`·도메인 존재 수준의 느슨한 검증(예: `^[^\s@]+@[^\s@]+\.[^\s@]+$`) | 과도한 정규식은 유효 주소를 거부 — 관례 |
| 잘림 방지 | `maxlength`로 이메일을 자르지 않는다 | 긴 유효 주소 존재 |

### 검증 실패 카피 (원인+해결)

| 상황 | 카피 |
|---|---|
| 이메일 형식 | 이메일 형식이 올바르지 않아요. @를 포함한 전체 주소를 입력해 주세요. |
| 이메일 중복 | 이미 가입된 이메일이에요. 로그인하거나 비밀번호 찾기를 이용해 주세요. |
| 비밀번호 조건 미달 | 비밀번호가 너무 짧아요. 8자 이상으로 입력해 주세요. |
| 비밀번호 불일치(확인 필드 사용 시) | 두 비밀번호가 서로 달라요. 같은 비밀번호를 다시 입력해 주세요. |
| 로그인 실패 | 이메일 또는 비밀번호가 일치하지 않아요. 다시 확인하거나 비밀번호 찾기를 이용해 주세요. |

로그인 실패는 이메일/비밀번호 중 무엇이 틀렸는지 특정하지 않는다. 근거: 계정 존재 여부 노출 방지(열거 공격).

## 7. 조사 안전 라벨 패턴

변수(이름·항목명) 뒤에 조사(은/는·이/가·을/를·와/과·(으)로)가 붙으면 받침 유무에 따라 깨진다. 라벨·다이얼로그 카피는 **조사가 변수에 닿지 않는 구조**로 쓴다.

| 위험 패턴 | 안전 재작성 |
|---|---|
| `{{name}}을 삭제할까요?` | `선택한 항목을 삭제할까요?` |
| `{{name}}이 추가되었습니다` | `추가 완료: {{name}}` |
| `{{name}}과 연결` | `연결 대상: {{name}}` / `"{{name}}" 연결` |
| `{{team}}으로 이동할까요?` | `이동할 팀: {{team}}` / `"{{team}}" 팀으로 이동할까요?` (뒤에 일반명사를 붙여 조사를 명사에 건다) |

- 따옴표로 변수 격리(`"{{name}}" 삭제`), 콜론 나열(`삭제할 항목: {{name}}`), 일반명사 후행(`{{team}} 팀`)의 3가지 패턴으로 대부분 해결된다. 근거: korean-ux-writing 변수·조사 절.
- 재작성으로 해결이 안 되는 동적 문장은 받침 기반 **동적 조사 헬퍼**를 쓴다 — `korean-number-date-units.md` §7의 `josa()` 헬퍼 참조.
