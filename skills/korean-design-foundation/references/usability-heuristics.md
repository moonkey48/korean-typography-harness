# 사용성 휴리스틱 (Nielsen 10) — 웹/B2B 적용 SSOT

한글 디자인 클러스터의 **사용성 판정 기준**. qa-checklist.md와 동급 인용 권위를 갖는다. 각 휴리스틱은 ①한글 명칭+정의 ②웹/B2B 적용 규칙(근거 병기) ③합격/불합격 예시 ④자동화 플래그로 구성한다. 타이포·간격·대비 수치는 여기서 새로 정의하지 않고 foundation 해당 파일(type-scale.md, spacing-grid.md, color-contrast.md)을 인용한다.

**자동화 플래그 범례**

| 플래그 | 의미 | 판정 주체 |
|---|---|---|
| 자동 | lint·axe·Playwright로 기계 판정 가능 | korean-design-checks (P0-7 규칙) |
| 부분 | 기계가 후보 검출, 사람이 최종 판정 | 기계 검출 + 검수자 확정 |
| 수동 | 사람 판정만 가능 | 체크리스트에 명시적 PASS/FAIL 기록 |

## H1. 시스템 상태 가시성

시스템이 지금 무엇을 하고 있는지, 사용자의 행동이 접수됐는지를 즉시 알린다.

| 규칙 | 근거 |
|---|---|
| 모든 인터랙션은 100ms 내 시각 피드백, 1초 초과 작업은 로딩 표시, 10초 초과는 진행률+취소 | NN/g 응답시간 한계(0.1/1/10초) |
| 제출 버튼은 로딩→성공/실패 상태로 전환하고 진행 중 재클릭(이중 제출)을 차단 | HIG/MD submit-feedback 웹 번안 |
| 토스트는 3~5초 자동 소멸 — 단, 중요 내용·필수 행동은 자동 소멸 금지(명시적 닫기 제공), 포커스를 뺏지 않고 `aria-live="polite"`로 알림 | 3~5초: MD 관례(toast-dismiss) 웹 번안 · aria-live: WCAG 4.1.3 상태 메시지 |
| 멀티스텝 플로우는 단계 표시기(현재/전체)를 상시 노출 | MD multi-step-progress 웹 번안 |
| 맥락 없는 스피너 금지 — 무엇이 로딩 중인지 명시("견적 계산 중…") | DBW INTERACTION-DESIGN |

- **합격**: 저장 클릭 → 버튼이 "저장 중…" 스피너로 바뀌고 완료 시 "저장됨" 토스트.
- **불합격**: 클릭 후 아무 변화 없다가 3초 뒤 갑자기 화면 전환. 그 사이 두 번 더 클릭됨.
- **자동화**: 부분 — 이중 제출 방지·로딩 상태 존재는 Playwright 검사(P0-7), 피드백 문구 적절성은 수동.

## H2. 현실 세계와의 일치

시스템 내부 용어가 아니라 사용자의 언어와 한국 표기 관례로 말한다.

| 규칙 | 근거 |
|---|---|
| 라벨·메뉴는 사용자 어휘로 — 내부 용어·DB 컬럼명·개발 용어 노출 금지 | NN/g H2 |
| 버튼은 결과를 서술 — "제출" 대신 "견적서 보내기", "확인" 대신 동사 라벨 | DBW USABILITY(information scent) |
| 날짜·숫자·단위·통화는 한국 표기 관례를 따르고 하드코딩하지 않는다(상세 규칙은 korean-number-date-units.md, P0-4) | 관례 |
| 동적 문장에서 조사(은/는·이/가·을/를)가 어긋나는 조합 금지 — 조사 안전 문형으로 재작성 | 관례 |
| 아이콘 단독 사용은 의미가 보편 관례인 경우(휴지통=삭제)로 한정, 그 외 텍스트 병기 | NN/g 아이콘 사용성 |

- **합격**: "2026.07.03 결제 예정 · 125만 원"
- **불합격**: "Jul 3, 2026 · TXN_STATE: PENDING · 1250000"
- **자동화**: 부분 — 하드코딩 날짜/숫자 포맷은 lint 검출(P0-7), 어휘·조사 적절성은 수동(+`korean-skills:humanizer` 패스).

## H3. 사용자 통제와 자유

실수했을 때 되돌릴 수 있고, 어떤 흐름에서든 탈출구가 있다.

| 규칙 | 근거 |
|---|---|
| 파괴적·대량 액션에는 undo 제공("삭제됨 · 실행 취소" 토스트) — 부록 A 사다리 1순위 | Apple HIG undo-support, NN/g H3 |
| 모달·시트는 명시적 닫기 버튼 + ESC 키로 닫힘 | HIG modal-escape 웹 번안, ARIA APG |
| 브라우저 뒤로가기는 예측대로 동작하고 스크롤·필터·입력 상태를 보존 | HIG/MD back-behavior·state-preservation 웹 번안 |
| 멀티스텝은 이전 단계로 돌아갈 수 있어야 한다 — 막다른 길 금지 | MD multi-step-progress |
| 저장 안 된 변경이 있는 화면·모달 이탈 시에만 확인을 요구 | HIG sheet-dismiss-confirm 웹 번안 |

- **합격**: 행 삭제 → 즉시 목록에서 사라지고 "삭제됨 · 실행 취소" 토스트 5초.
- **불합격**: 3단계 결제 플로우에서 뒤로가기를 누르면 입력 전체가 사라지고 홈으로 이동.
- **자동화**: 부분 — undo/확인 UI 존재·멀티스텝 뒤로가기는 Playwright(P0-7), 상태 보존 범위는 수동.

## H4. 일관성과 표준

같은 의미는 같은 모양·같은 말·같은 위치로. 외부 관례를 깨지 않는다.

| 규칙 | 근거 |
|---|---|
| 내부 일관성: 동일 액션은 모든 화면에서 동일 라벨·컴포넌트·위치 | NN/g H4 |
| 외부 관례: 로고 클릭=홈, 헤더 또는 좌측 내비, 데스크톱에서 주 내비를 숨기지 않기 | DBW USABILITY, 관례(Jakob의 법칙) |
| 인터랙티브/비인터랙티브 시각 구분 — 링크는 링크처럼, 클릭 안 되는 텍스트를 링크처럼 꾸미지 않기 | DBW INTERACTION-DESIGN(affordance) |
| 타이포·간격·색은 토큰 체계만 사용 — 수치와 역할은 type-scale.md·spacing-grid.md·color-contrast.md 인용 | foundation SSOT |
| 데스크톱(≥1024px)은 사이드바, 좁은 화면은 상단 내비로 전환 — breakpoint 수치는 spacing-grid.md | MD adaptive-navigation 웹 번안 |

- **합격**: 모든 화면의 기본 저장 버튼이 동일 컴포넌트·동일 라벨("저장").
- **불합격**: 화면마다 "확인/저장/완료/등록"이 혼용되고 색과 위치도 제각각.
- **자동화**: 부분 — 토큰 이탈은 stylelint(korean-design-checks), 라벨·위치 일관성은 수동.

## H5. 오류 예방

오류 메시지를 잘 쓰는 것보다 오류가 날 수 없게 설계하는 것이 먼저다.

| 규칙 | 근거 |
|---|---|
| 3단 방어 순서: ①불가능하게(입력 제약·피커·값 추론) ②어렵게(스마트 기본값·미리보기) ③회복 쉽게(undo) | DBW INTERACTION-DESIGN, NN/g H5 |
| 파괴적 액션은 부록 A의 확인 사다리(undo>소프트삭제>미리보기>타이핑 확인)를 적용 — 확인 모달 남발 금지 | NN/g 확인 피로 연구 |
| semantic input type(email/tel/number)과 자동 포맷(전화번호 하이픈)을 기본 적용 — 한국 폼 상세는 korean-forms.md(P0-2) | HIG/MD input-type 웹 번안 |
| 긴 폼은 초안 자동 저장으로 이탈·새로고침 시 데이터 유실 방지 | HIG form-autosave 웹 번안 |
| 불가능한 액션은 비활성화하되 이유를 함께 표시(툴팁·helper text) | DBW INTERACTION-DESIGN |

- **합격**: 워크스페이스 삭제는 이름 타이핑 확인 + 30일 소프트 삭제로 이중 방어.
- **불합격**: 사소한 액션마다 "정말 하시겠습니까?" 모달 — 정작 영구 삭제는 확인 없이 실행.
- **자동화**: 부분 — 파괴적 액션 확인/undo 부재는 Playwright(P0-7), 방어 수준의 적정성은 수동.

## H6. 회상보다 재인

기억을 요구하지 말고, 필요한 정보와 선택지를 화면에 보여준다.

| 규칙 | 근거 |
|---|---|
| 옵션·형식·맥락을 화면에 노출 — 이전 화면 내용을 외워야 진행되는 흐름 금지 | NN/g H6 |
| 선택지 10개를 넘는 드롭다운은 검색/typeahead로 대체 | Baymard 드롭다운 연구 |
| 현재 위치 표시: 내비 활성 상태 필수, 3단계 이상 깊이는 breadcrumb | NN/g breadcrumb, MD breadcrumb-web |
| 검색에는 최근 검색어·추천어를 제시 | MD search-accessible 웹 번안 |
| 형식 안내는 placeholder가 아닌 지속 표시 helper text로 — placeholder는 입력 시작 시 사라진다 | NN/g placeholder 유해성 연구, MD input-helper-text |

- **합격**: 사업자등록번호 필드 아래 "숫자 10자리, 하이픈 자동 입력" helper text 상시 노출.
- **불합격**: placeholder에만 형식 안내 → 한 글자 입력하면 사라져 형식을 기억해야 함.
- **자동화**: 부분 — placeholder-only 라벨은 axe/lint 검출, breadcrumb·재인 설계 적정성은 수동.

## H7. 유연성과 효율 (웹 B2B 보강)

초보자를 방해하지 않으면서 숙련자에게 가속 장치를 제공한다. B2B 도구는 같은 화면을 하루 수십 번 여는 사용자가 기준이다.

| 규칙 | 근거 |
|---|---|
| 빈도 높은 과업에 키보드 단축키 제공 + 단축키 안내 노출(`?` 키 목록 또는 툴팁 병기) | NN/g accelerator(H7 원문), 관례(Gmail·Slack·Linear) |
| 화면 수가 많은 도구(대략 5개 이상)는 커맨드 팔레트(Cmd/Ctrl+K)가 관례 | 관례(Notion·Linear·Slack) |
| 목록의 필터·정렬·검색 상태를 URL에 반영 — 새로고침 유지·링크 공유·저장된 뷰의 기반 | MD deep-linking 웹 번안, 관례(B2B 어드민) |
| 표는 다중 선택 + 일괄 액션(삭제·상태 변경·내보내기)을 제공하고 undo와 결합 | 관례(B2B), Apple HIG undo-support |
| 자주 쓰는 값을 기억: 최근 사용 항목·마지막 선택을 기본값으로 | NN/g H7, DBW(smart defaults) |

- **합격**: 필터 걸린 목록 URL을 동료에게 보내면 같은 화면이 열린다.
- **불합격**: 새로고침하면 필터·페이지가 초기화되고, 반복 작업에 단축 경로가 전혀 없다.
- **자동화**: 수동 — URL 상태 보존만 Playwright로 부분 검사 가능.

## H8. 미학적이고 최소한의 디자인

화면의 모든 요소는 주 과업과 경쟁한다. 경쟁자를 줄인다.

| 규칙 | 근거 |
|---|---|
| 화면당 주 액션 1개 — 가장 두드러진 인터랙티브 요소는 주 CTA여야 한다 | DBW USABILITY(시각 계층) |
| 5단 판독 순서 확보: 페이지 목적 → 주 액션 → 주 내용 → 보조 액션 → 부가 정보 | DBW USABILITY |
| 장식 요소·중복 CTA·의미 없는 애니메이션 제거 — 애니메이션은 상태/피드백 의미가 있을 때만 | DBW USABILITY(noise) |
| 고급 옵션은 점진적 노출(접기·"더보기") — 첫 화면에 전부 펼치지 않기 | HIG progressive-disclosure 웹 번안 |
| 밀도·행 길이·스케일 수치는 새로 정하지 않는다 — type-scale.md, design-type-matrix.md 인용 | foundation SSOT |

- **합격**: 대시보드 상단에 핵심 지표 4개, 상세는 클릭 드릴다운.
- **불합격**: 첫 화면에 설정 30개·버튼 8종·배너 3개가 동시에 주의를 다툰다.
- **자동화**: 수동 — 부록 C의 10초 글랜스 테스트로 판정.

## H9. 오류 인지·진단·회복 지원

오류가 났을 때 사용자가 스스로 이해하고 고칠 수 있게 한다.

| 규칙 | 근거 |
|---|---|
| 에러 카피 3부 구성: ①무엇이 실패했나 ②원인 ③다음 행동 — "결제에 실패했어요. 카드 한도 초과예요. 다른 카드로 다시 시도해 주세요." | ux-writing 스킬, NN/g 에러 메시지 가이드 |
| 인라인 에러는 해당 필드 바로 아래 + `aria-describedby`로 필드와 프로그램적 연결 | WCAG 3.3.1, MD error-placement |
| 제출 실패 시 첫 에러 필드로 포커스 이동, 에러가 여럿이면 상단 요약 + 각 필드 앵커 링크. **요약 발동 조건: 필드 6개 이상 폼에서 에러 2개 이상** — 그 미만은 인라인+포커스 이동으로 충분 (F5) | WCAG 2.4.3·3.3.3, MD focus-management |
| 에러 후 사용자 입력을 지우지 않는다 — 입력 보존은 회복 비용을 결정한다 | DBW INTERACTION-DESIGN, Baymard 폼 연구 |
| 비난 어투·기술 코드 단독 노출·"알 수 없는 오류" 금지, 한글 카피 어조는 korean-content-design 규칙 적용 | ux-writing 스킬, 관례 |
| 에러 표시는 색만으로 하지 않기(아이콘/텍스트 병기), 대비 기준 수치는 color-contrast.md | WCAG 1.4.1·1.4.3 |

- **합격**: "이메일에 @가 필요해요" — 필드 아래 표시, 포커스 이동, 입력 유지.
- **불합격**: 페이지 상단에 "오류가 발생했습니다 (code: 0x80070005)" — 어느 필드인지 불명, 입력 초기화.
- **자동화**: 부분 — 필드 연결·focus-to-error·aria-live는 axe/Playwright(P0-7), 카피 품질은 수동.

## H10. 도움말과 문서 (웹 B2B 보강)

도움말이 필요 없는 것이 최선이지만, 필요하다면 과업 맥락 안에서 즉시 닿아야 한다.

| 규칙 | 근거 |
|---|---|
| 도움말은 맥락 안에: 복잡한 필드·설정 옆 ⓘ 툴팁 또는 한 줄 설명 — 별도 매뉴얼로 보내지 않기 | NN/g 맥락 도움말(proactive help) |
| 빈 상태는 교육 기회: 왜 비었는지 + 첫 행동 CTA + 필요 시 예시 데이터 | ux-writing 스킬(empty states), 관례 |
| 온보딩은 과업 기반: 기능 나열 투어 대신 첫 성공 과업으로 안내, 건너뛰기 상시 허용 | NN/g 온보딩 투어 연구(대다수 사용자가 skip) |
| 문서는 전문 검색 가능해야 하고, 제품 화면에서 해당 문서 섹션으로 딥링크 진입 | 관례(B2B — Stripe·Linear 문서 패턴) |
| 같은 질문이 반복되면 문서가 아니라 UI 결함 신호 — 설계로 역류시킨다 | NN/g H10 원칙 |

- **합격**: 권한 설정의 각 역할 옆 ⓘ 툴팁이 역할 차이를 한 줄로 설명.
- **불합격**: 도움말이 별도 PDF 매뉴얼에만 존재하고 제품 안에서 진입 경로가 없다.
- **자동화**: 수동.

## 부록 A. 파괴적 액션 확인 사다리

약한 단계부터 검토한다. 아래로 갈수록 마찰이 커지므로, 되돌릴 수 있는 액션에 강한 확인을 걸면 확인 피로로 실효를 잃는다(NN/g).

| 단계 | 패턴 | 적용 기준 | 근거 |
|---|---|---|---|
| 1. undo | 확인 없이 즉시 실행 + "실행 취소" 토스트(5초 내외) | 되돌릴 수 있는 모든 삭제·이동·상태 변경 | Apple HIG, NN/g |
| 2. 소프트 삭제 | 휴지통/보관함으로 이동, 유예 기간(예: 30일) 후 영구 삭제, 복구 UI 제공 | 사용자 생성 콘텐츠·문서·항목 | 관례(Gmail·Notion·Slack) |
| 3. 미리보기 확인 | 영향 범위를 구체 수치로 표시("항목 128개가 삭제됩니다") 후 확인 | 대량 작업, 연쇄 영향이 있는 작업 | DBW INTERACTION-DESIGN |
| 4. 타이핑 확인 | 대상 이름을 직접 입력해야 실행 | 되돌릴 수 없고 영향이 큰 것만(계정·프로젝트·저장소 삭제) | 관례(GitHub·AWS) |

확인 카피 규칙: 대상을 명명하고, 영향을 서술하고, 안전한 탈출구를 준다. 버튼 라벨은 동사("삭제하기"/"취소") — "확인/취소" 쌍 금지 (DBW, ux-writing).

## 부록 B. 폼 검증 타이밍 (blur + submit)

| 시점 | 동작 | 근거 |
|---|---|---|
| 입력 중(keystroke) | 오류 검증 금지. 예외: 글자수 카운터, 아이디 중복 등 가용성 조회 | MD inline-validation, Baymard 조기 검증 연구 |
| blur(필드 이탈) | 형식 검증 실행, 에러는 필드 바로 아래 표시 | MD, Baymard |
| 에러 상태에서 재입력 | 수정 시작 즉시 재검증하여 통과하면 에러 제거 — "보상은 빨리, 벌은 늦게" | Baymard, 관례 |
| submit | 전체 검증 + 첫 에러 필드 포커스 + 에러 다수 시 상단 요약(H9) | WCAG 2.4.3·3.3.3 |
| 서버 검증 | 비동기 결과도 같은 위치·같은 형식으로 표시 — 인라인/서버 에러의 표시 이원화 금지 | 관례 |
| 제출 중 | 버튼 로딩 상태 + 이중 제출 차단(H1) | HIG/MD 웹 번안 |

한국 특화 입력(휴대폰 3-4-4·주소검색·사업자등록번호 체크섬)은 korean-forms.md(P0-2)가 정본.

## 부록 C. IA·온보딩·체감 성능 요점

**IA (정보 구조)**
- trunk test 5문: 임의의 내부 페이지에서 ①무슨 사이트인가 ②무슨 페이지인가 ③주요 섹션은 ④내 위치는 ⑤검색은 어디에 — 즉답 불가면 사이트 ID·페이지명·내비 구조부터 고친다 (Krug, DBW USABILITY 번안).
- 내비 3질문: 어디에 있나 / 여기에 뭐가 있나 / 어디로 갈 수 있나 (DBW USABILITY).
- 상위 섹션은 화면 간 고정, 페이지명은 클릭한 링크 문구와 일치 (DBW USABILITY).
- 라벨은 기발함보다 명백함 — 사용자가 이미 아는 이름을 쓴다 (DBW USABILITY).

**온보딩**
- 첫 실행은 기능 투어보다 빈 상태 + 첫 과업 CTA가 낫다 (NN/g 온보딩 연구, H10).
- 가입 시 필수 입력 최소화 — 나머지는 사용 중 점진 수집 (Baymard 폼 필드 연구).

**체감 성능**
- 0.1초/1초/10초 한계: 0.1초는 즉각으로 지각, 1초까지 흐름 유지, 초과 시 진행 표시 필수 (NN/g).
- 구조가 예측되는 로딩은 스켈레톤, 짧은 불확정 대기는 스피너 (관례).
- 낙관적 업데이트는 실패 시 롤백 UI(토스트+복원)가 있을 때만 (관례).
- LCP 등 수치 게이트는 korean-design-checks 소관 — 이 문서는 지각 기준만 정의.
