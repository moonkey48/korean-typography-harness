# Audit Rubric

## Finding Format

Use this exact logic for every issue:

| Field | Required content |
|---|---|
| Severity | critical, major, minor |
| Location | file/component/route/screenshot area |
| Symptom | visible or source-level evidence |
| Human read | why a person reads it as Bamti |
| Fix | specific correction |
| Confirmation needed | yes only for deletion, full layout rewrite, brand font replacement, CTA change, or proof removal |

## Content Triage

Before redesigning, classify content:

| Bucket | Action |
|---|---|
| Required fact | preserve and improve hierarchy |
| Primary action | keep one dominant action per screen |
| Optional support | demote or group |
| Decorative copy | remove unless it clarifies a decision |
| Proof claim | keep only if user/source supplied it |
| Redundant section | deletion candidate; confirm if user-visible |
| Risky deletion | ask the user before removing |

## Review Questions

- Can a user state what this is, who it is for, and what to do next in 5-10 seconds?
- Does the first screen show product/domain specificity, not layout presets?
- Are Korean headings, labels, buttons, and links readable at mobile widths?
- Are state, focus, disabled, loading, empty, and error behaviors real?
- Would the UI still work if decorative visuals were removed?
